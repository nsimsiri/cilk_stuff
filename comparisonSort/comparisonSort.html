Comparison Sort (SORT):

Sort items of any (uniform) type using any comparison operation (less-than) that defines a total order. The input can have equal elements defined by the comparison (i.e. items such that neither a < b nor b < a), but this does not mean the items are the same. The sort need not be stable. The code must not take advantage of the specific type beyond the comparison operator supplied by the type and possibly the size of the key. If the size of the key is used by the program then the benchmark should be indicated with -Z (SORT-Z).

Input and Output File Formats

The input and output need to be in the sequence file format. The output file must be in sorted order with respect to the approrpriate type.

Default Input Distributions

Each distribution should be run for n=10,000,000. The weights used for average time are given in parentheses.
  • (1) Double precision floating point numbers from the uniform distribution.
    randomSeq -t double <n> <filename>
  • (1) Double precision floating point numbers from the exponential distribution.
    exptSeq -t double <n> <filename>
  • (1) Double precision floating point numbers from an almost-sorted distribution.
    almostSortedSeq -t double <n> <filename>
  • (3) Strings from a tri-gram distribution.
    trigramSeq <n> <filename>
  • (3) Strings from tri-gram distribution, but order randomized after allocation. In particular after the data is read in, the pointers to the strings have to be randomly shuffled (not included in timing). The purpose of this test is to see how the sort does when strings are not allocated adjacently with respect to the input pointers.
    trigramSeq <n> <filename>
